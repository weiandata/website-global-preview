# A25 · Merging Small Sample Units: Robust Handling of Stratified Proportional Statistics

> A deep dive into survey statistics. General method explanation, not involving specific tools or products.

## A Blind Spot in All Stratified Reporting

Survey reports must present the proportion of population units by "province × urban/rural × gender × age group" (e.g., the distribution of a literacy level A/B/C/D). The table is designed beautifully, until you see the data: some units have only 7 samples, and some are even 0.

A "D level" of 14.3% calculated from 7 samples is actually "1 person"; 0-sample units have no numbers at all. **Publishing such tables is equivalent to printing sampling noise as regional differences.** This issue exists in any detailed report, but most teams either avoid it (by cutting out the detailed breakdown dimensions) or handle it dangerously (by quietly filling in numbers from neighboring units).

## The Proper Approach: Merging—But With Constraints

The correct solution for small units is **merging**: group units with insufficient samples with "similar" units, and estimate the proportion using the combined sample. It sounds simple, but doing it right is difficult, as merging must satisfy a set of mutually restrictive requirements:

1. **Each final group must have a sufficient sample size** (e.g., ≥30), especially **zero-sample units must never be grouped alone**—they must be merged into a group with samples, otherwise their proportion estimate is made up from nothing;
2. **Similarity**: Merged units should be similar in population structure (grouping 20-year-old urban males with 65-year-old rural females results in a group with no meaningful mean);
3. **Consistency constraint**: After merging, the estimated proportions should align with the target interval when aggregated back to the provincial or national level (aligned with the overall and historical reporting definitions);
4. **Minimal distortion**: Merging should cause as little distortion to the weight structure and as little erasure of each unit's "own voice" as possible;
5. **Reproducibility**: The same input must yield the same merging scheme—when a report is questioned, "how was this group formed" must have a definite answer.

## Why This Is a Combinatorial Optimization Problem

Each of the above points seems manageable on its own, but together they form a classic **constrained combinatorial optimization** problem: dividing N units into several groups (with an exponential number of possible divisions), and optimizing under hard constraints such as sample size, connectivity/similarity, and consistency intervals, according to a priority order (e.g., minimizing: number of violated constraints → heterogeneity within groups → distortion of weights → number of merges).

Manual handling of such problems leads to predictable outcomes: trial and error until late at night, satisfying one constraint while breaking another, and redoing everything when data updates, with no one able to prove that the current solution is not just a coincidence. The proper approach is to hand it over to **Mixed Integer Linear Programming (MILP)**: constraints are written as linear inequalities, and the priority order is implemented using lexicographic objectives. Mature solvers (such as the open-source HiGHS) provide an **optimal and deterministic** solution. After solving, run an independent audit process to verify each group's sample size and each constraint—what the solver says is satisfied doesn't count; what the audit script says is satisfied does.

## Professional Ethics in This Context

There are two "easy" approaches that must be explicitly rejected:

**Imputing Data**: Filling zero-sample units directly with the national average or neighboring province numbers. This is not estimation—it's fabrication. Every number in a publication should be able to answer "based on which observations."

**Pretending Feasibility**: When constraints are contradictory (e.g., a province's overall sample size is insufficient to support the target interval), force a "looks like it satisfies" solution. The correct behavior is to **explicitly report structural infeasibility**: inform decision-makers that "these constraints cannot all be satisfied with this data," and provide quantified relaxation suggestions (how much to relax each interval to make it feasible). **The credibility of statistical work is precisely built on the courage to say "it's not possible."**

By the way: such "honest failures" should be built into the processing pipeline—stop and report when infeasibility is detected, rather than silently outputting the closest solution and letting users mistakenly believe it meets the criteria.

## A Roadmap for Practitioners

1. Conduct **capacity estimation** during the design phase of the report: based on expected sample sizes, which breakdown levels can be sustained? Those that cannot either expand the sample or cut dimensions (aligning with the "minimum reporting unit principle" from the sample size section);
2. After data collection: conduct a census of unit sample sizes → define merging rules (constraint list + priority order) → model and solve → independent audit → document the solution;
3. Publish the merging solution along with the technical report: which units were merged into groups, and the sample size of each group—transparency is the best defense against questioning;
4. Maintain stable merging rules across years (changing rules introduces methodological noise into the "change").

The small unit problem is a typical example in the intersection of "survey statistics × optimization": trivial in business, interesting in mathematics, unnoticed when done correctly, and newsworthy when done wrong. This kind of work—translating reporting definitions into constraints and trade-offs into objective functions—is the daily routine of professional data processing.
