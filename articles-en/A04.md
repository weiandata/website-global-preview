## Why Build Another IRT Package

There are already excellent IRT packages in the R ecosystem, such as mirt and TAM. Why build another one? The answer lies in a real engineering pain point: **when the number of respondents reaches hundreds of thousands or even millions, and the number of items is in the hundreds with multi-dimensional models, traditional implementations run into memory issues.**

The E-step of IRT estimation (marginal maximum likelihood, MML) requires computing the posterior probability for "each respondent × each integration node." For a million respondents and a multi-dimensional model, the intermediate matrix can easily reach hundreds of gigabytes—this is not just slow, it's simply too large to fit in memory. Large-scale literacy assessments and national surveys are precisely at this scale.

IRTC is designed for this purpose: **a self-contained MML estimation engine with C++ (Rcpp/RcppArmadillo) as the backend, with systematic performance optimizations for large-scale data.** It is currently open-sourced on GitHub (search for IRTC or visit my homepage), version v0.1.x, MIT-style open-source license, with 26 sets of automated tests and a Chinese beginner's manual.

Let's clarify the positioning to help you decide whether to continue reading:

- **Conventional analysis for a few thousand to a few ten thousand respondents**: mirt is a great choice. There is plenty of documentation and a mature ecosystem, so there's no need to switch.
- **Over 100,000 respondents, memory warnings, or the need to run multi-dimensional models at the million level on a regular workstation**: IRTC is designed for you.

## What It Can Do

**Model Coverage**: Rasch/1PL, 2PL, PCM, RSM, GPCM—mainstream models for dichotomous and polytomous scoring; unidimensional and multidimensional between items; supports latent regression (background variables in the model), multi-group analysis, case weights, and EAP individual ability estimation. This list covers most modeling needs for large-scale assessments.

**Performance Design** (this is where the differentiation lies):

1. **Multi-threaded parallel E-step**: Naturally parallelizable at the respondent level, accelerating with multiple cores.
2. **Streaming posterior computation**: Process respondents in blocks, compute and discard the posterior immediately—**avoiding the storage of the full "respondent × node" matrix**. Memory usage shifts from being proportional to the sample size to being proportional to the block size, which is the key to running million-level samples on regular workstations.
3. **Dimension decomposition**: Decompose high-dimensional integrals in multidimensional item structures to reduce dimensionality, avoiding exponential growth in the number of nodes with increasing dimensions.
4. **Analytical gradient + hybrid Newton M-step**: Fewer and more stable iterations.
5. **Automatic engine routing**: The `auto` mode automatically selects between grid (conventional) or streaming (large-scale) paths based on data size, and records the rationale for the selection.

**Trust Mechanisms**: Optional integration node pruning for faster computation includes layered empirical validation of approximation errors and outputs a report. By default, it follows the exact path—**acceleration never silently changes the statistical target**. This is a rule we set for ourselves.

## Ten-Minute Getting Started

```r
# Installation (GitHub)
remotes::install_github("makunxiang-cmd/IRTC")
library(IRTC)

# Data: A respondent × item response matrix (0/1 or 0,1,2,... for polytomous scoring)
# Simulated data for demonstration; replace with your matrix in practice
dat <- sim_irt_data(n = 5000, k = 40, model = "2PL")  # Based on repository documentation

# Fitting a 2PL model, engine automatically routes
fit <- irtc(dat, model = "2PL", engine = "auto")

summary(fit)          # Convergence information, item parameter overview
coef(fit)             # Difficulty/discrimination parameter table
theta <- ability(fit) # EAP ability estimate and standard error
```

Typical configuration for large-scale data:

```r
fit <- irtc(dat_big, model = "GPCM", dims = 3,
            engine = "streaming",   # Explicitly using the streaming engine
            threads = 8,            # Parallel threads
            weights = w,            # Sampling weights
            covariates = X)         # Latent regression background variables
```

## A Usage Recommendation: Start Small, Then Scale Up

No matter which package you use, the correct approach for large-scale modeling is: **first draw a subsample of 1–50,000 respondents, solve all model specifications, item diagnostics, and convergence issues, and confirm the solution before running the full dataset.** A full run can take several hours, while debugging can be done in minutes on a subsample. The project timeline can differ by an order of magnitude. IRTC's results are consistent with mainstream implementations at conventional scales (this is part of the 26 sets of automated tests). Therefore, "which package to use for the subsample" is entirely flexible.
