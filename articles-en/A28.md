# A28 · Why Is Statistical Software Trustworthy: Automated Testing and "Honest Failures"

> Engineering quality line. Differentiated concept output, suitable for submission to the Statistical China rewrite version.

## An Uncomfortable Fact

Errors in statistical software almost never manifest as crashes. Instead, they appear as: **a number that looks completely normal, quietly being wrong.**

A missing weight multiplied once less, missing values treated as zero, a too loose convergence criterion, or a formula error under a certain boundary condition — these errors do not throw exceptions or pop up windows. They quietly slip into reports, papers, and policy bases. Users have no way of noticing: who can judge with their eyes whether "0.437" should be "0.512"?

Therefore, the question "Is this statistical tool trustworthy?" is a question that must be answered seriously. This article provides a practical answer framework — it is both our internal standard when developing tools (such as the open-source IRTC), and can be directly used as a checklist for evaluating any statistical software (including when assessing the code accompanying a paper during peer review).

## Four Pillars of Trustworthiness

### Pillar One: Benchmarking Verification — Comparing with Known Answers

**Method:** Hand-calculated results from textbook examples, widely accepted outputs from classic datasets, and estimates from mature software are compared one by one within tolerance limits.

This is the most straightforward and effective step. **An implementation that has never been compared with external answers is only correct according to the author's own feeling.** For users, this means checking whether there are benchmarking records in its documentation or tests; for developers, it means that each model should be compared with at least one cross-implementation benchmark before going live.

### Pillar Two: Simulation Recovery — The Golden Test of Statistical Correctness

Statistical methods have a luxurious self-checking mechanism: **generating data on your own.** Generate simulated data with known true parameters → estimate using the software → check whether the estimates are distributed around the true values in the correct way (bias approaching zero, standard error calibration, confidence interval coverage at the nominal level).

This step is stronger than benchmarking: benchmarking can only prove "being the same as others," while simulation recovery proves "statistical correctness under model assumptions." It can also answer "at what sample size does this method become stable" — this is the empirical source of the "recommended sample size" in the documentation.

### Pillar Three: Boundary and Fail-Safe — Errors Should Die at the Door

Real data inevitably brings boundary conditions: empty categories, all-correct or all-wrong items, constant variables, single-record datasets, and illegal encodings. Trustworthy software has clear behavior for each of these: **either handle them correctly or clearly report an error** — the worst behavior is to silently provide a number.

In testing, this corresponds to "boundary test cases"; in design, it corresponds to "entry validation + clear error messages." When evaluating a package, intentionally feed it bad data and observe its response — the information from this five-minute test exceeds reading ten pages of promotional material.

### Pillar Four: Regression Protection — Fixed Bugs Should Not Be Repeated

Each fixed bug is solidified as a permanent test; every code change automatically reruns all tests (CI, continuous integration). This ensures that the software's trustworthiness is **monotonically non-decreasing** — without regression protection, every "optimization" is like rolling a dice.

From an evaluation perspective: whether a GitHub repository has a test directory and a CI badge is the most direct signal of engineering literacy.

## A Higher Standard: "Honest Failures"

Above the four pillars, there is a design philosophy we call "honest failures":

- **Explicitly Displaying Capability Limits:** When a calculation is not possible (structural infeasibility, non-convergence, insufficient sample size to support the model), clearly state "it cannot be calculated" and the reason, never silently downgrade or output the closest guess.
- **Approximations Require Informed Consent:** Any option that trades precision for speed (such as integral pruning or approximation algorithms) should be default disabled, explicitly enabled, and accompanied by an error verification report — users should always know whether they are getting an exact solution or an approximation.
- **Truthfully Conveying Uncertainty:** Do not quietly narrow wide confidence intervals; do not hide convergence warnings in logs.

The opposite of this philosophy is something everyone has seen: software that can produce results for any input and never alerts for any situation. **In the short term, it's about user experience, but in the long term, it's transferring all methodological risks to the person least capable of identifying them.**

## Action Checklist for Three Types of Readers

**Software Users:** Prioritize tools with testing, CI, and benchmarking records; cross-validate critical results with an independent implementation.

**Code Authors (Researchers):** Write at least three types of assertions for your analysis scripts — data assertions (dimension, range, sample size conservation), known answer anchors, and boundary error checks. Using testthat or ten lines of `stopifnot` is fine; there's something more important than perfection.

**Purchasers:** Include "delivery includes automated testing" in the contract. Especially in scenarios where the program needs to run long-term in your environment (such as the "data never leaves the premises" delivery model), testing is remote quality assurance — this is our default configuration for all deliveries, and it should also become your default requirement for any supplier.

The trustworthiness of statistical software has never been a function of the author's reputation or a polished interface, but rather a function of this process. If the process exists, trust exists.
