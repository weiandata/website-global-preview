## A Common Deadlock

Numerous research institutions, government departments, and public institutions in China hold high-value survey data — such as public opinion surveys, literacy assessments, and industry censuses. These data share two common characteristics: **the professional threshold for analysis is high** (measurement modeling, sampling weighting, equating and linking are all specialized methodologies), and **the data itself cannot leave the premises** (due to personal information and sensitive domains, with clear institutional red lines).

This creates a deadlock: internal teams lack sufficient personnel or methodological expertise, while external experts have the capability but cannot access the data. The three common "solutions" each come with their own costs —

- **Trying to do it ourselves**: The methodological quality of the results is uncertain, and the final output is vulnerable to scrutiny.
- **Bending the rules to release the data**: This carries massive compliance risks, and a data leak would be a deal-breaker.
- **Abandoning in-depth analysis**: Spending millions on data that only yields a few descriptive statistics tables is a waste of resources.

In reality, there is a fourth path, and it has been successfully implemented for years: **let the tools enter, not the data.**

## The Core of the Model: Tool and Data Isolation

When breaking down the "analysis service," what is truly irreplaceable for external experts? It is **methodological design and program implementation**, not the act of "touching the data." Based on this insight, the delivery process can be restructured as follows:

**Step 1: Develop based on structure, not data.**
The client provides a data dictionary (variable list, types, value ranges, sample frame description) — optionally with a small number of fully anonymized sample records. The vendor then **generates simulated data with a structure that is completely consistent**: same variables, same encoding, same magnitude, and every value is fabricated.

**Step 2: Complete all development and testing on the simulated data.**
Data cleaning, modeling, weighting, and report generation — the entire analysis pipeline is developed, debugged, and tested on the simulated data. Automated testing is included to assert sample size, weight sums, and parameter value ranges — ensuring the program is not just "runnable" but "correct."

**Step 3: The program enters, the client runs it.**
Deliver a complete set of analysis programs and an operation manual. The client's own staff runs the program on an internal network or isolated environment using real data, producing **aggregated results** (parameter tables, statistical reports, and visualizations).

**Step 4: Remote iteration based on anonymized outputs.**
Real data will always have situations that simulated data cannot predict (new missing patterns, extreme values, encoding exceptions). The program's design should convert these into **diagnostic logs without individual data** — allowing the vendor to identify and iterate on issues by reviewing the logs and aggregated outputs.

Throughout the process, the external team never touches any real data record. **The physical boundary and management responsibility of the data remain unchanged** — this is not "a safer outsourcing" but the elimination of data leaving the premises at its root.

## What This Model Means for the Client

**Compliance-wise**, the clause "the vendor does not access the raw data" can be directly written into the contract, making it stricter than any confidentiality agreement — while confidentiality agreements govern "not leaking data once obtained," this model ensures "data is never obtained in the first place."

**In terms of assets**, traditional outsourcing delivers a report (a one-time use item), while this model delivers **a reusable program** — data can be rerun next year, and minor adjustments in reporting definitions can be handled by changing the configuration. For multi-cycle monitoring projects, the marginal cost drops sharply from the second year, and the institution builds up its own operational capability.

**In terms of quality**, this model actually drives higher engineering standards. Since the program must be run in an environment unseen by the vendor and by non-developers, it must be: **reproducible** (fixed seed and version, same input leads to same output), **self-validated** (automated testing built-in), and **honest about errors** (explicitly stating when it cannot handle a situation, rather than silently outputting suspicious results). In other words, the constraint of "data never leaves premises" eliminates the "just run it and hand it over" workshop-style delivery — such ad-hoc scripts simply cannot survive in this model.

## When This Model Applies and When It Needs Adaptation

**Applicable scenarios**: Analysis with clear objectives for repetitive or procedural tasks (modeling and weighting for annual monitoring, standard reporting), and one-time in-depth analysis with clearly defined methods.

**Adaptation needed**: Highly exploratory analysis (where the data's content is unknown, and the direction is determined as you go). When pure program delivery is insufficient, you can combine **on-site work** (vendor personnel visiting the client's premises, using the client's equipment for exploration, with results retained by the client) or **secure room/remote desktop** (controlled access in an isolated environment), and then return to programmatic delivery once exploration is complete. The three levels of isolation can be paired according to the sensitivity level of the data.

## A Checklist for Institutions Selecting This Model

1. Clearly state in the contract that "the vendor does not access the raw data" and the list of deliverables (programs, tests, manuals, codebook);
2. Set the acceptance criteria as "the client's staff independently runs the full process according to the manual";
3. Require the program to include automated testing and version tracking — this is the lifeline for troubleshooting in the internal network;
4. Reserve 2–3 rounds of "real data feedback — version iteration" in the schedule (simulated data cannot predict everything);
5. Evaluate the vendor based on publicly accessible engineering works (open-source projects, technical documentation). The program delivery model requires a much higher level of engineering capability from the vendor than traditional consulting.
