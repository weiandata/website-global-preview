# A15 · Missing Data Handling: A Complete Map from Deletion to Multiple Imputation

> A deep dive companion to Q13. Core long article on data cleaning.

## Missingness is Information, Not Garbage

Beginners treat missing data as garbage to be cleaned, while professionals analyze missingness as **information**—who didn't answer, what they didn't answer, and why they didn't answer. The root cause of choosing the wrong handling method is almost always not answering the question "why is it missing" first.

## Step 1: The Three-Classification of Missing Mechanisms (Rubin's Framework)

**MCAR (Missing Completely at Random)**: The probability of missingness is unrelated to any variables (e.g., a page missing from a printed questionnaire). At this point, any reasonable method is unbiased, and deletion only loses efficiency. Little's MCAR test can be used for verification, but a non-significant result ≠ proof of MCAR (due to power issues), and it can only be used as a reference.

**MAR (Missing at Random)**: The probability of missingness depends on **observed** variables (e.g., older people are less likely to answer internet-related questions, and age is known). This is the working assumption for modern methods (MI, FIML). Note that MAR cannot be directly tested—it is an assumption about unobserved data and can only be indirectly supported through "analysis of the association between missingness and observed variables."

**MNAR (Missing Not at Random)**: The probability of missingness depends on the missing value itself (e.g., higher income individuals are more likely to refuse to answer income questions). All conventional methods are biased under MNAR. The proper approach is sensitivity analysis (selection model / pattern mixture, or simply recalculating conclusions under different extreme assumptions to check stability).

**Practical Recommendation**: Make "missing mechanism analysis" a routine report section—missing rates for each variable, missing pattern plots (`mice::md.pattern`, naniar package), and a table of the association between missing indicators and key observed variables. This one page of analysis gives your method selection a solid basis.

## Step 2: Method Map

**Deletion Methods**:

- **Listwise deletion**: Only acceptable when missingness is low (<5%) and close to MCAR; biased under MAR, and in multivariate analysis, "missing a little on each variable" can accumulate to delete half the sample;
- **Pairwise deletion**: Different statistics are based on different subsets, and the correlation matrix may be non-positive definite, which is dangerous in SEM—avoid using it.

**Single-Value Imputation** (Mean/Regression/Hot Deck): The common flaw is **treating imputed values as true values**, which underestimates variance, makes confidence intervals too narrow, and p-values too small. Mean imputation also compresses the distribution and distorts correlations. Except for low-stakes scenarios such as "missing in auxiliary variables," it is not recommended.

**Multiple Imputation (MI)—The General Workhorse**:
Logic in three steps: impute m times (each imputation set includes random perturbations, honestly reflecting uncertainty) → analyze each imputation set separately → combine using Rubin's rules (point estimate is the mean, variance = within + between—between variance is precisely the "additional uncertainty from missingness").
Practical Tips (R: mice package):

1. Take m ≥ 20+ (old textbooks say 5, modern standards adjust based on missing rate);
2. **The imputation model should be wider than the analysis model**: all variables used in analysis (including the dependent variable, interaction terms) should be included in the imputation model, otherwise the corresponding relationships are pulled toward zero;
3. Diagnose after imputation: compare the distribution of imputed values with the observed values (density plots);
4. Use appropriate methods for ordered or categorical variables (polr/logreg), don't treat them all as continuous.

**FIML (Full Information Maximum Likelihood)—The Workhorse for SEM Scenarios**:
No imputation, directly use the available information from each individual in the likelihood. In lavaan, `missing = "ML"` completes the task in one parameter. Under MAR, FIML is asymptotically equivalent to MI. For CFA/SEM, prioritize FIML (avoiding the imputation-merging process), and use MI when reporting descriptive statistics or using non-SEM methods.

## Step 3: Special Rules for Test Data

Missingness in ability assessments has a unique classification, and handling conventions differ:

- **Omitted**: The item was skipped. Common handling: treat as missing during item parameter calibration, and handle as incorrect during ability estimation (or partial scoring), with each item having its own convention. The key is to **explicitly state the rules**;
- **Not Reached**: Time ran out at the end of the test. During calibration, treat as missing (otherwise, difficult item parameters are contaminated by "slow responders"), and the handling during ability estimation should be clearly stated in the technical report;
- **Planned Missingness** (matrix test construction): Missingness caused by design, naturally MAR (missingness is entirely determined by "which test booklet the item is assigned to"), and IRT/PV frameworks automatically handle it correctly—this is the classic contrast of "planned missingness is harmless, behavior-based missingness is problematic."

## Final Checklist

- [ ] Report missing rates, patterns, and mechanism analysis
- [ ] Use MI or FIML for the main analysis; deletion methods are only for sensitivity checks
- [ ] The imputation model should cover all variables used in the analysis model
- [ ] If suspecting MNAR: compare the robustness of conclusions under different assumptions
- [ ] For test data: encode omitted and not reached separately, and document the handling rules

The level of missing data handling can immediately reveal a team's statistical literacy—this cannot be faked by default software settings; every step is an explicit method decision.
