# A23 · Reproducible Research Workflow: The Complete Recipe of R + Git + Automated Testing

> Companion to Q24's in-depth version. Engineering quality line.

## A Hidden Illness in the Industry

"Last month's chart can't be reproduced anymore," "The result changed after switching computers," "Students graduated, but no one knows how to handle this data" — every data-intensive team has these stories. The reproducibility crisis is not just an academic ethics issue; it is first and foremost **engineering debt**: analyses that are not reproducible lose all value the moment they are questioned, audited, or handed over.

The good news is that the cure is cheap. Below, I follow the structure of "solving one type of accident per layer" to provide a complete recipe.

## Layer One: Scripting — Eliminate Manual Traces

**Accident Type**: "I remember deleting a few rows in Excel back then..."

Rule: **Raw data is only read and frozen; no manual steps are allowed from raw data to the final chart.** All cleaning, transformation, modeling, and plotting must be done via code. The test standard is: delete all intermediate files, and the entire analysis should be reconstructable with the script alone.

Companion habits: Project template directory structure (`data-raw/` (read-only), `R/`, `output/`), relative paths + `here` package (absolute paths are the reason for failure when switching computers).

## Layer Two: Git — Give Your Project a Time Machine

**Accident Type**: `analysis_final_v3_true_final(modified).R`

Git solves "version archaeology": each logical change is a commit, and any historical state is traceable. Researchers can use it in a minimal way — just three actions, `add / commit / push`, recover 80% of the value. Pair with a private GitHub/Gitee repository to solve both backup and collaboration.

Commit discipline is more important than the tool: write "why you changed" in the commit message ("Fix weight truncation threshold: the original value led to low ESS"), and you will be the biggest beneficiary three months from now.

## Layer Three: Lock Randomness and Environment — Eliminate "Schrödinger's Results"

**Accident Type**: "The same code produces different results for him and me."

Lock down three sources one by one:

1. **Random Numbers**: Explicitly `set.seed()` for all random steps (sampling, imputation, simulation, some optimizers). Random number generation in parallel computing is a separate pitfall: use `RNGkind("L'Ecuyer-CMRG")` or the built-in parallel-safe seed in the future framework;
2. **Package Version**: `renv::init()` + `renv::snapshot()` generates a lock file, and a new environment `renv::restore()` precisely reconstructs it. Accidents where R package major versions change default behaviors are real (some packages have changed their default encoding and aggregation behavior);
3. **System Environment**: A lightweight approach is to archive `sessionInfo()`; for heavy needs (cross-year audit, cluster deployment), use Docker containers to freeze the operating system as well.

## Layer Four: Pipeline — Make "Re-running Everything" a One-Click Operation

**Accident Type**: "I changed the cleaning rules, but forgot which steps to re-run..."

Manually running seven scripts in sequence will inevitably lead to an error in order or a missed step. The solution is a pipeline tool: the **targets package** declares dependencies between steps, and it automatically identifies and re-calculates only the affected downstream steps when any upstream is changed. The byproduct is the entire analysis becoming a visualized dependency graph — this itself is the best process documentation.

A simplified alternative: a `run_all.R` master script sources all steps in sequence, although it lacks incremental computation, it achieves the baseline of "one-click re-run."

## Layer Five: Automated Testing — From "Can Re-run" to "Dare Re-run"

**Accident Type**: "Refactored a function, and the result changed quietly, but no one noticed."

Write assertions for "facts that must hold" (testthat or lightweight `stopifnot`):

- Data assertions: sample size conservation, ID uniqueness, weight sum equals population, proportions within [0,1];
- Result anchors: core tables match the frozen version (snapshot testing);
- Function unit tests: known input-output pairs for self-written functions.

With testing, "bold refactoring and safe re-running" becomes possible. All analysis programs we deliver to clients are fully equipped with tests — the programs must run in environments unseen by us, by non-developers ("data never leaves the premises" mode), and the tests are the remote guarantee of quality.

## Layer Six: Text and Code as One Source — Automated Reporting

**Accident Type**: The numbers in the paper don't match the code output (manual transcription errors).

Quarto/R Markdown writes narrative and computation in the same document: all numbers, tables, and figures are generated by code blocks, and the entire text is re-rendered when data updates, automatically refreshing all numbers. **There are no manually entered statistics in the report**, and the species of transcription errors is extinct.

## Adoption Roadmap (Don't Overdo It)

- Do this this week: Layer One (Scripting) + Layer Two (Git) — the lowest cost and highest return;
- Starting with the next project: Layer Three (renv + seeds) + Layer Six (Quarto);
- For teams / long-term projects: Layers Four and Five (targets + testing).

Initially, each task is 20% slower, but after three projects, the net gain is achieved due to a sharp reduction in rework. Reproducibility is not a moral burden — it is **an insurance policy for your future self and colleagues** — and in our field, it is already the entry ticket to professional delivery.
