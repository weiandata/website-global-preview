# A21 · A Quality Control System for Survey Data: The Full Lifecycle from Design to Release

> Practical assessment operations. A systematic long article for project managers, with high business value.

## Quality Is Not Inspected Into Being

A principle understood a century ago in manufacturing, often forgotten in the survey industry: **Quality is the product of design and process, and end-of-line inspection can only intercept, not create.** For a survey with tens of thousands of samples, if quality control only occurs "after the data comes back," the only option left is to perform a last-minute fix. This article presents a QC framework covering the entire lifecycle — it is also an operational version of the "Total Survey Error" framework.

## Phase One: Design Stage — Blocking Each Source of Error

**Measurement Error Defense Line**: Construct definition → expert review → cognitive interviews → pilot testing, each process has deliverables (see the scale development section). **Skipping this week will cost a month later.**

**Coverage and Sampling Error Defense Line**: Checking the completeness of the sampling frame (people not in the frame will never be sampled, this is a coverage error that no weighting can fix); matching the sampling plan with the target precision (planning by the smallest reporting unit, see the sample size section); **each level of sampling probability must be documented on-site** — this is the raw material for weighting.

**Quality Checkpoints**: Instruction verification questions, contradictory pairs, and timestamp records — detection methods must be embedded before the questionnaire is finalized.

**Plan Freezing**: Cleaning rules, exclusion criteria, weighting routes, and core analysis are written into a "Data Processing Plan" and reviewed before the data arrives. This is both a systemic barrier against p-hacking and a construction blueprint for multi-person collaboration.

## Phase Two: Data Collection Stage — Real-Time Monitoring, Not Post-Hoc Surprises

The biggest benefit of electronic data collection is **real-time visibility of the process data**. Use it as follows:

- **Dual Monitoring of Progress and Structure**: Not only monitor the number of responses, but also the response structure (completion rates of each layer/unit), intervene immediately if a certain unit systematically lags, rather than relying on weighting to pull it up after the fact;
- **Daily Quality Reports**: Distribution of new sample durations, straight-line rates, and logical conflict rates, grouped by collection point/fieldworker — address abnormal clusters of collection points (e.g., a fieldworker's questionnaire completed on average in 3 minutes) on the same day;
- **On-Site Verification Mechanism**: Randomly select a proportion for re-verification (have they actually visited? Are key answers consistent?), a traditional rule for household surveys, which also applies to online surveys (device fingerprinting, IP clustering detection);
- **Military Rule for Mid-Project Revisions**: Any change requires version registration and synchronized updates to the new-old coding mapping table. Uncontrolled questionnaire revisions are one of the leading causes of data disasters in large projects.

## Phase Three: Data Processing Stage — Six Steps of Cleaning + Weighting + Full Traceability

The six steps of cleaning (structure validation → quality screening → missing data triage → coding derivation → external consistency → cleaning report) and the three-stage weighting (design weight → non-response adjustment → calibration) have been detailed in previous sections. From a systemic perspective, we add three more:

1. **Dual Calculation**: Key steps (weights, core indicators) are independently implemented by two people, with results cross-checked. Although costly, it's worth it for formal projects — it blocks the most expensive errors;
2. **Code as Documentation**: Full scripting + version control, "Data Processing Notes" are automatically generated from scripts rather than manually written (manually written documentation inevitably diverges from actual code);
3. **Asserted Intermediate Outputs**: Each output includes automated checks (sample size conservation, weight sum equals population, proportions within [0,1]), and any assertion failure stops the process. All processing programs we deliver to clients are embedded with such assertions — these assertions act as remote eyes when the program runs in the client's environment.

## Phase Four: Analysis and Release Stage — The Final Gate

- **Consistency Audit of Definitions**: Every number in the report can be traced back to a unique generation script; cross-check definitions between technical reports and press releases (θ scale vs. correct response rate, weighted vs. unweighted), as inconsistent definitions are a classic script for public criticism after release;
- **Final Inspection of Extreme Values**: Scan all summary tables for intuitive magnitude checks (e.g., a province's mean is 3 standard deviations higher than its neighbor? Check the data before discussing findings);
- **Sensitivity Appendix**: Robustness of key conclusions under "different cleaning rules / different weighting settings," which is much more dignified to write proactively than to patch after being questioned;
- **Reproducible Archiving**: Freeze the data version + code + environment list + technical documentation. Goal: After three years, audit and reproduce with one click.

## A Health Check Sheet

Score your project (1 point per item):

1. There is a written data processing plan that arrives before the data
2. Sampling probabilities are documented at each level
3. There are daily quality reports and intervention records during data collection
4. Cleaning rules are frozen before analysis
5. Raw data is read-only, and the entire process is script-based
6. Weights have complete technical documentation
7. Key numbers are double-checked with dual calculation
8. Report numbers can be traced back with one click
9. There is sensitivity analysis
10. The entire project is archived for reproducibility

Projects scoring above 8 are serious projects; projects scoring below 5 should only be used for internal reference. This system has been refined over many years in our national large-scale surveys — it's not mysterious, but the difficulty lies in truly doing each item.
