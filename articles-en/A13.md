# A13 · Test Equating and Linking: Making Scores from Different Tests Speak the Same Language

> A deeper version of Q16. Core technology for continuous monitoring projects.

## Comparability is Designed

The legality of any cross-test, cross-year comparison is based on a technical premise: the two sets of scores have been placed on the same ruler. This process is called equating/linking. This article explains its design, methods, and engineering considerations — and why **equating is a design phase task, and it's too late to think about it after data collection**.

## Premise: There Must Be a Bridge

Two tests administered to two separate groups of people with no overlap make equating mathematically impossible. Three bridge designs:

**1. Anchor Item Design (NEAT, most commonly used)**: Two tests share a set of anchor items. Engineering requirements:

- **Anchor Item Quantity**: At least 20% of the total test items (less than this, linking errors dominate everything);
- **Representativeness**: The content distribution and difficulty distribution of anchor items should be a reflection of the entire test (using only medium-difficulty items as anchors leads to inaccuracies at the extremes);
- **Position Stability**: The position of the same anchor item should be as consistent as possible across the two tests (position effects can masquerade as difficulty drift);
- **Confidentiality**: **Leakage of anchor items = equating failure = trend conclusion failure**.

**2. Common Group Design**: The same group of people takes two tests (or randomly cross-administered halves). Suitable for small-scale calibration, fatigue and order effects need to be controlled with balanced designs.

**3. Random Group Design**: The same population is randomly assigned to different tests. Logically the cleanest, but the execution quality of "random assignment" is required to be extremely high.

## Methods: Two Major Families

**Classical Equating** (at the level of observed scores):

- **Mean Equating / Linear Equating**: Align the means (and standard deviations) of the distributions, simple and straightforward;
- **Percentile Equating**: Match the percentiles of the two tests (if 78 points is the 63rd percentile in Test A, it maps to the 63rd percentile score in Test B), flexible but requires large sample smoothing.
Classical methods are intuitive and make no model assumptions, but are sensitive to sample representativeness and cannot provide "item-level" diagnostics.

**IRT Equating** (at the level of parameters) — mainstream for large-scale assessments:

The core is to place the item parameters of the two tests on the same scale, with three routes:

1. **Fixed Anchor Parameters**: During new test calibration, anchor item parameters are fixed to the existing values in the item bank, and new item parameters automatically fall onto the existing scale. This is the routine practice for item bank operations;
2. **Separate Calibration + Transformation**: Each test is calibrated separately, then a linear transformation coefficient is estimated using anchor item parameters (mean-mean, mean-standard deviation, or more robust Stocking-Lord / Haebara characteristic curve methods — the latter is less sensitive to individual abnormal anchor items and is the recommended default);
3. **Concurrent Calibration**: Merge the data from both tests and estimate parameters in one go, handling missing responses according to the design. Statistically the most efficient, but one bad apple spoils the whole batch (any issue in one test's data contaminates the whole set), large projects commonly use "concurrent calibration as the main method + separate calibration for cross-validation".

R Tools: equate (classical methods), plink/SNSequate (IRT transformation), mirt/TAM (multi-group concurrent calibration).

## Engineering Considerations: Where Equating Most Often Fails

**1. Anchor Item Drift.** Anchor item parameters significantly change between two administrations — possible reasons include curriculum reform (content becomes more familiar), leakage, or position changes. **Drift detection must be done before equating** (parameter difference test / displacement plot), and drifting anchor items should be excluded from the equating set. What if there are not enough anchor items after exclusion? There is no good solution — so redundancy must be designed in advance.

**2. Accumulation of Linking Errors.** A→B→C→D annual linking, with error accumulation at each step, it's unclear how much linking noise is mixed into the trend after five years. Countermeasures: design a "back-anchoring" mechanism (directly link to a baseline year every few years), and report linking errors (include equating uncertainty in the standard error of trend inference — the "linking error" item in large-scale assessment reports is exactly this).

**3. Construct Drift.** The meaning of "numeracy literacy" has changed over the past decade, and the items have become outdated — at this point, even if statistical equating is perfect, "score comparability" in substantive terms is already questionable. Long-term projects should regularly conduct framework revisions and bridging studies (administer both old and new frameworks in one round), which goes beyond statistical techniques and is a project governance issue.

## Checklist for Project Leads

- [ ] Before administration: Anchor item layout design (quantity / representativeness / position) + confidentiality plan
- [ ] During calibration: Anchor item drift detection → exclusion → transformation method (default Stocking-Lord) + sensitivity comparison
- [ ] When reporting: Linking error included in trend inference; equating method and anchor item handling fully documented in technical documentation
- [ ] Long-term: Back-anchoring design + bridging plan for framework revision

Equating is a classic "good work goes unnoticed, bad work ruins everything" behind-the-scenes engineering task. The credibility of a continuous monitoring project is half dependent on sampling weighting and half on equating.
