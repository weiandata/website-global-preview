## Why Researchers Should Learn to Write Packages

Every piece of analysis code you've written that you thought you'd use again faces the same fate at a crossroads: rotting in some project folder; being copied and pasted into seven slightly different variants; or — being packaged into a tool with documentation, tests, and installability.

The third path has a cost lower than most people imagine (modern toolchains have automated all mechanical labor), but the returns are compounding: **a package is the executable form of a research method**. It makes your method installable, verifiable, and citable, and forces you to think through the interface and assumptions. Below is the complete path from zero to release, with practical notes from our open-source IRTC package development process.

## Phase One: Skeleton and Minimum Viable (One Afternoon)

```r
usethis::create_package("mytool")   # Generate a standard skeleton
usethis::use_mit_license()          # Choose a license
usethis::use_git()                  # Start version control from day one
```

Move functions into `R/`, and write roxygen2 comments (`@param`, `@return`, `@examples`, `@export`) above each function, then:

```r
devtools::document()   # Comments → Help documentation
devtools::load_all()   # Load for debugging
devtools::check()      # Comprehensive check: 0 errors, 0 warnings as the goal
```

`check()` is your mentor: fix whatever it complains about, and after fixing, you'll understand the package standards. The only reference book needed throughout is Wickham's *R Packages* (free online).

## Phase Two: Make It Trustworthy (The Deciding Phase)

**Testing — the life line of statistical software.**
The danger of statistical code: it doesn't report errors when it calculates incorrectly; it just quietly gives a wrong number. Start with `usethis::use_testthat()`, and arrange your testing strategy by importance:

1. **Correctness Anchors**: Compare with known answers (hand-calculated small examples, textbook problems, results from mature software);
2. **Simulation Recovery**: Generate data with known true parameters → estimate → assert that the deviation is within tolerance (the golden test of statistical methods);
3. **Boundary and Fail-Safe Checks**: Empty data, all missing, single-row input, illegal parameters — each should result in **clear error messages** rather than mysterious crashes;
4. **Regression Testing**: Each fixed bug is solidified into a test, and the same pit is not fallen into twice.

Experience scale: testing code volume ≥ functional code volume is considered passing. In IRTC's 26 test groups, simulation recovery and cross-implementation alignment take up most — this is the foundation of "engine calculation being correct."

**Error Philosophy.** When input is invalid, report in plain language at the first site of the problem ("resp contains negative values, the response matrix should be non-negative integers"), rather than letting users dig through 20 layers of call stacks. **Explicitly reject** situations beyond the ability boundary, never silently provide suspicious results — "honest failure" is the virtue of statistical software.

**Vignette.** Function documentation answers "how to call," while a vignette answers "what scenarios, complete workflow, and how to interpret results." A good vignette often has more reach than the package itself.

## Phase Three: Performance (If Your Field Requires It)

First, squeeze out performance at the R level: vectorization, avoid growing objects inside loops, data.table. If that's not enough, move to Rcpp:

```r
usethis::use_rcpp_armadillo()
```

Move bottleneck functions (usually likelihood calculations, which are tight loops) down to C++, leaving R with only the interface. IRTC's architecture is exactly like this: the estimation engine is all C++ (including multithreading and streaming memory management), and users still face a regular R function. **Principle: Optimize only after profiling (profvis package), and optimization based on intuition is often wrong.**

## Phase Four: Release

**GitHub Release (can be completed on the same day):** Push to the repository + README (a short paragraph explaining what problem it solves + installation command + minimal example) + GitHub Actions running CI (`usethis::use_github_action("check-standard")`, automatic cross-platform check). Users can then install via `remotes::install_github()`. This is already a proper open-source release.

**CRAN Submission (optional advanced step):** The value is the trustworthy endorsement and reach via `install.packages()`. Additional work: policy compliance (example runtime limits, no writing to user disk, complete documentation), cross-platform pre-checks (`devtools::check_win_devel()`, rhub), and `cran-comments.md` explanation. Manual review, and being rejected on the first try is normal — just fix each issue and resubmit. Mindset: the reviewer is helping you reach the release quality.

**Citation and Attribution:** Add an `inst/CITATION` file, so users can automatically cite the author and source when citing in their papers — the long-term return of academic tools comes from here.

## Time Budget Reference

Minimum viable in 1 day; a trustworthy version with tests and vignette in 1–2 weeks (part-time); CRAN-level polishing adds another 1–2 weeks. Compared to the development of the method itself, the packaging cost is a small fraction, but it determines the **survival radius** of the method: scripts in a folder don't survive beyond the project cycle, but an open-source package with tests and documentation can live for many years.
