# A27 · 甲方指南：如何与外部数据分析团队高效合作

> 配套 Q26 的深度版。商务转化线，以"帮甲方避坑"立场建立信任。

## 写给需求方的实话

这篇文章站在你们（研究机构、政府部门、评价机构）的立场写：怎么把数据分析的外部合作做成，而不是做砸。内容来自乙方视角看到的真实翻车现场——**多数失败的合作，问题在启动前就埋下了**。

## 启动前：三个必须自问的问题

**1. 你买的到底是什么？**
"找人做数据分析"至少是四种不同的采购，错配是失败的头号原因：

- **买结果**：一次性把这批数据做完出报告 → 项目制；
- **买方法**：方案设计、技术把关、评审应对 → 顾问制（按天/按阶段）；
- **买工具**：每年重复的分析流程 → **程序交付**（对方交付你们自己能跑的软件资产）；
- **买能力**：让团队学会自己做 → 培训+陪跑。

特别提示第三种：**凡是"每年都要做一遍"的分析，买程序都比买人工划算**——首年成本相当，次年起边际成本趋近于零，而且能力沉淀在你机构。很多甲方不知道市场上存在这个选项。

**2. 数据能不能出门？**
涉个人信息、涉敏感领域的数据，这个问题要在联系任何供应商**之前**想清楚。数据不能出门 ≠ 不能合作——成熟的隔离模式（程序交付、现场作业、安全屋）完全可以支撑专业合作（我写过一篇《数据不出门》专门讲这套模式）。但它必须写进采购需求，因为**能做隔离交付的供应商是少数**，这一条会直接筛掉大半候选者，早筛早省事。

**3. 内部谁来对接？**
外部合作最被低估的成本是甲方自己的对接投入。必须指定一位**懂数据的内部负责人**（不必是统计专家，但要能回答"这个变量当时怎么采集的"），并预留其工时。没有这个角色，任何乙方都会在"猜数据含义"上烧掉一半工期。

## 选型：五个问题探出深浅

1. "我的抽样是分层整群的，你们怎么处理？"——不问你要抽样方案就报价的，直接淘汰；
2. "交付物包含可运行的代码吗？我们能复现吗？"——只交报告不交代码，你买到的是一次性烟花；
3. "为什么这个方法适合我的问题？"——听解释深度，专业方讲得出依据与局限，作坊只会"都是这么做的"；
4. "有什么公开可查的作品？"——论文、开源项目、技术文档。GitHub 上的代码质量骗不了人，嘴上的"服务过大项目"无法验证；
5. "如果分析中发现数据支撑不了预期结论，怎么办？"——**这是试金石**。正确答案是"如实报告"；任何暗示"能调出你们想要的结果"的，是在邀请你共谋，将来暴雷时机构信誉一起赔进去。

## 合同：把专业要求变成条款

建议写入的关键条款（每条都对应一类真实纠纷）：

- **交付物清单**：报告 + 全部分析代码 + 运行说明 + 数据字典/codebook（+ 程序交付模式下的自动化测试）；
- **验收标准**："甲方人员按说明在甲方环境复现主要结果"——这一条同时保证了代码质量和知识转移；
- **数据安全**：接触边界（谁、何种形式、何种环境）、留存与销毁、违约责任；隔离交付模式直接写明"乙方不接触原始数据"；
- **中期检查点**：至少一次中期交付（如清洗报告+初步描述），别把全部风险压到期末；
- **知识产权**：代码归属或使用授权，避免"明年想改个参数还要付一次全款"。

## 协作期：让专业者做专业判断，但保持知情

- 开工先开**数据交底会**：采集背景、已知的坑、口径变迁——你多说一小时，项目少走一星期弯路；
- 尊重方法建议，但要求**所有方法选择有书面理由**（技术备忘录），这既是质量控制也是你的资产；
- 对"坏消息"给出奖励性反应：乙方报告"这批数据撑不起省级结论"是专业负责的表现，逼他"想想办法"是在逼他造假；
- 期末除了验收结果，做一次**移交演练**：你的人从零跑通全流程。跑不通，验收不算完。
