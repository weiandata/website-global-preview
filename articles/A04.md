## 为什么又造一个 IRT 包

R 生态里已有 mirt、TAM 这些优秀的 IRT 包，为什么还要自研？答案是一个真实的工程痛点：**当被试规模到数十万、上百万，题目上百、模型多维时，传统实现会在内存上撞墙。**

IRT 估计（边际极大似然，MML）的 E 步需要计算"每个被试 × 每个积分节点"的后验概率。百万被试 × 多维模型的积分网格，这个中间矩阵动辄数百 GB——不是算得慢，是根本装不下。而大型素养监测、全国性调查恰恰就是这个量级。

IRTC 就是为此设计的：**一个自包含的 MML 估计引擎，C++（Rcpp/RcppArmadillo）底层，针对大规模数据做了系统性性能设计**，目前在 GitHub 开源（搜索 IRTC 或从我主页进入），v0.1.x，MIT 类开源协议，配 26 组自动化测试与中文零基础手册。

先说清楚定位，帮你决定要不要往下读：

- **几千、几万被试的常规分析**：用 mirt 就很好，资料多、生态熟，不必换；
- **数十万以上被试、内存开始报警、或需要在普通工作站跑百万级多维模型**：IRTC 是为你准备的。

## 它能做什么

**模型覆盖**：Rasch/1PL、2PL、PCM、RSM、GPCM——二级与多级计分主流模型；单维与题间多维；支持潜在回归（背景变量进模型）、多组分析、个案权重、EAP 个人能力估计。这个清单覆盖了大规模测评的绝大多数建模需求。

**性能设计**（这是差异化所在）：

1. **多线程并行 E 步**：被试层面天然可并行，多核直接加速；
2. **流式后验计算**：按被试分块，后验算完即用即弃，**不落地完整的"被试 × 节点"矩阵**——内存占用从"正比于样本量"变成"正比于块大小"，这是百万级样本能在普通工作站上跑的关键；
3. **维度分解**：题间多维结构下把高维积分拆解降维，避免节点数随维度指数爆炸；
4. **解析梯度 + 混合 Newton M 步**：迭代更少更稳；
5. **自动引擎路由**：`auto` 模式根据数据规模自动选择 grid（常规）或 streaming（大规模）路径，并记录选择理由。

**可信赖机制**：可选的积分节点剪枝加速会对近似误差做分层实测核查并输出报告；默认走精确路径——**加速永远不悄悄改变统计目标**，这是我们给自己定的规矩。

## 十分钟上手

```r
# 安装（GitHub）
remotes::install_github("makunxiang-cmd/IRTC")
library(IRTC)

# 数据：被试 × 题目的作答矩阵（0/1 或 0,1,2,... 多级计分）
# 演示用模拟数据，实际使用换成你的矩阵
dat <- sim_irt_data(n = 5000, k = 40, model = "2PL")  # 以仓库文档为准

# 拟合 2PL，引擎自动路由
fit <- irtc(dat, model = "2PL", engine = "auto")

summary(fit)          # 收敛信息、题目参数概览
coef(fit)             # 难度/区分度参数表
theta <- ability(fit) # EAP 能力估计与标准误
```

大规模数据的典型配置：

```r
fit <- irtc(dat_big, model = "GPCM", dims = 3,
            engine = "streaming",   # 显式走流式引擎
            threads = 8,            # 并行线程
            weights = w,            # 抽样权重
            covariates = X)         # 潜在回归背景变量
```

（函数名与参数以 GitHub 仓库最新文档为准，仓库内有完整的中文手册和可复制的示例脚本。）

## 一个使用建议：先小后大

不管用哪个包，大规模建模的正确姿势都是：**先抽样一个 1–5 万的子样本，把模型设定、题目诊断、收敛问题全部解决，确认方案后再上全量。**全量跑一次几个小时，调试阶段在子样本上分钟级迭代，工期差出一个量级。IRTC 的结果在常规规模上与主流实现对标一致（这正是那 26 组自动化测试的一部分），所以"子样本用什么包验证"完全灵活。
