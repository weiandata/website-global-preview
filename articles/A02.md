# A02 · 大规模问卷数据清洗：完整流程与那些让项目翻车的坑

> 手册首批主题②（实操向）。一鱼四吃：知乎专栏 → 公众号 → B站实操视频 → 小红书"避坑清单"卡片。

## 为什么清洗值得写一篇长文

在大规模调查这个行当里流传一句话：**分析的上限，在数据交到你手上那一刻就定了；清洗决定你能不能摸到这个上限。**

几十万份问卷的数据集里，藏着乱答的、答一半的、逻辑矛盾的、被采集系统编码错的、被中途改版问卷坑的。这些问题不清理，后面的建模越精致，错得越体面。本文给出一套在数十万样本量级的全国性调查中反复验证过的六步流程，以及每一步真实翻过的车。

## 第零步：原则先行

**原始数据是文物：只读、存档、永不修改。**

所有清洗动作都写成脚本，从原始数据出发生成分析数据集。这带来三个救命的性质：任何一步可回溯；规则可以被审查；发现清洗错误时，改一行代码重跑即可，而不是对着改过八轮的 Excel 考古。

直接在 Excel 里手工删行改值，是调查数据处理的第一大忌——三个月后没有任何人（包括你自己）能说清这份数据经历过什么。

## 第一步：结构校验——先确认"数据长得对"

拿到数据先别看内容，先做结构断言：

- 样本量与现场报的回收量对得上吗？
- ID 唯一吗？（重复 ID 背后常是采集端重传，两条记录可能一条完整一条残缺）
- 变量数量、类型、取值范围与问卷设计一致吗？（5 点量表里出现 6，是真实数据错误还是"拒答"编码？）
- **跳答逻辑成立吗？** 答"没有上过网"的人，后面不该有"每天上网时长"。

这一步全部写成自动化检查脚本，逐条输出通过/不通过。**翻车案例原型**：问卷中途改版，某题选项从 4 个变成 5 个，两版数据混在同一列——不做取值范围校验，这种问题会一路潜伏到报告发布。

## 第二步：作答质量筛查——清理"人"的问题

电子化调查中低质量作答占 5%–15% 是文献反复确认的常态。检测指标按性价比排序：

1. **作答时长**：总时长 + 关键页面时长。阈值定法可用"中位数的三分之一"或绝对底线（如每题不足 2 秒）；
2. **长串指标**：连续选择同一选项的最大长度（直线作答）；
3. **逻辑矛盾对**：问卷里语义互斥的题目组合（最好在设计阶段就有意埋设）；
4. **规律模式**：ABABAB、12345 循环；
5. **个体内一致性**：同维度题目分半相关，认真作答者应当高相关；
6. 如果做了 IRT 建模，**人拟合指标**（lz 等）能再捞出一批作答模式异常者。

**关键纪律**：单一指标不定罪。每个指标都会误伤（深思熟虑的人作答慢，真实态度极端的人看起来像直线作答）。成熟做法是多指标标记 + 规则组合（如命中两条及以上进入复核/剔除），并且**规则在看数据之前定好、写进方案**——否则"哪种删法让结果显著就用哪种"，你就在 p-hacking。

## 第三步：缺失分诊——三种"空"是三种问题

- **合法跳答**：问卷逻辑决定的空，不是缺失，编码时与真缺失严格区分；
- **拒答/漏答**：题目级缺失，按缺失机制处理（低比例近随机可删；否则多重插补或 FIML）；
- **未及**：能力测验里时间不够没做到的题，与"跳过"机制不同，大规模测评的惯例是分别编码、模型中差别处理——统统记 0 分会系统性冤枉做题慢的人。

**翻车案例原型**：拒答编码 99 没转 NA，直接进了均值计算，某地区"平均每周锻炼 11 小时"——所有极端汇总值都值得先怀疑缺失编码。

## 第四步：编码与派生变量

反向计分（只转一次！转换后改列名防重复执行）、开放题编码、维度分计算、年龄段等分组变量生成。**每个派生变量在代码手册（codebook）里记三件事：来源变量、生成规则、生成脚本位置。** codebook 是大型项目里被低估最狠的交付物——它是数据的说明书，没有它，数据在人员流动一次之后就成了黑箱。

## 第五步：外部一致性校验

清洗完的数据要和外部基准对一对：样本的性别年龄结构与抽样框偏差多大（为加权做准备）？关键指标和上一轮调查/权威统计的量级对得上吗？出现 3 倍的跳变，先怀疑数据而不是急着解读"社会剧变"。

## 第六步：清洗报告——让每一条数据的去向可交代

一张流程表：进多少 → 每条规则删多少（占比）→ 最终分析样本多少。附各规则定义与阈值依据。正式项目这是技术报告必备章节，学位论文这是外审和答辩的护身符。讲究一点，再补一个敏感性分析：剔除前后核心结论是否稳定。

## 规模上来之后的工程问题

十万级以上样本，清洗本身成了计算任务：Excel 直接出局；R 里用 data.table/arrow 处理千万行毫无压力，但要遵守"筛选聚合下推、中间对象及时清理"的内存纪律。更重要的是**全流程自动化**——大项目的数据往往分批到达、多次补传，手工流程做三批就崩溃，脚本化流程只是重跑一次的事。

## 收尾：把清洗当成正式阶段

给它预算工期（经验值：占整个分析周期的 30%–50%）、给它交付物（清洗脚本 + codebook + 清洗报告）、给它验收标准（结构断言全部通过、剔除规则可辩护）。数据质量这件事，事前的一分严谨，胜过事后的十分补救。
