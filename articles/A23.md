# A23 · 可复现研究工作流：R + Git + 自动化测试的完整配方

> 配套 Q24 的深度版。工程质量线。

## 一个行业的暗病

"三个月前的图表现在跑不出来了""换了台电脑结果变了""学生毕业了，没人知道这份数据怎么处理的"——每个数据密集型团队都有这些故事。可复现性危机不只是学术伦理话题，它首先是**工程债**：不可复现的分析在被质疑、被审计、被交接的那一刻，价值归零。

好消息是解药便宜。以下按"每一层解决一类事故"的结构，给出完整配方。

## 第一层：脚本化——消灭手的痕迹

**事故类型**："我记得当时在 Excel 里删过几行……"

铁律：**原始数据只读冻结；从原始数据到最终图表之间，不允许任何手工步骤。**所有清洗、变换、建模、绘图都是代码。检验标准：删掉全部中间文件，能用脚本完整重建。

配套习惯：项目模板化目录（`data-raw/`（只读）、`R/`、`output/`），相对路径 + `here` 包（绝对路径是换电脑必死的原因）。

## 第二层：Git——给项目装时间机器

**事故类型**：`分析_final_v3_真最终(修改版).R`

Git 解决"版本考古"：每个逻辑改动一次提交，任何历史状态可回溯。研究者用法可以极简——只要 `add / commit / push` 三个动作就收回 80% 的价值。配 GitHub/Gitee 私有仓库，同时解决备份与协作。

提交纪律比工具更重要：提交信息写"为什么改"（"修正权重截断阈值：原值导致 ESS 过低"），三个月后的你是最大受益人。

## 第三层：锁定随机性与环境——消灭"薛定谔的结果"

**事故类型**："同样的代码，他跑出来和我不一样。"

三个来源逐个锁死：

1. **随机数**：一切随机步骤（抽样、插补、模拟、某些优化器）显式 `set.seed()`。并行计算的随机数是独立的坑：用 `RNGkind("L'Ecuyer-CMRG")` 或 future 框架的内建并行安全种子；
2. **包版本**：`renv::init()` + `renv::snapshot()` 生成锁文件，新环境 `renv::restore()` 精确还原。R 包大版本改默认行为的事故真实存在（某些包的默认对照编码、聚合行为都变过）；
3. **系统环境**：轻量做法是存档 `sessionInfo()`；重度需求（跨年审计、集群部署）用 Docker 容器把操作系统一起冻结。

## 第四层：流水线化——让"重跑全部"成为一键操作

**事故类型**："改了清洗规则，但忘了重跑后面哪几步……"

手工按顺序跑七个脚本，必然有一天顺序错、漏一步。解法是流水线工具：**targets 包**声明步骤间的依赖关系，改动任何上游，它自动识别并只重算受影响的下游。副产品是整个分析变成一张可视化的依赖图——这本身就是最好的流程文档。

简化版替代：一个 `run_all.R` 主控脚本按序 source 全部步骤，虽无增量计算，但"一键重演"的底线达成了。

## 第五层：自动化测试——从"能重跑"到"敢重跑"

**事故类型**：重构了函数，结果悄悄变了，没人发现。

给"必须成立的事实"写断言（testthat 或轻量的 `stopifnot`）：

- 数据断言：样本量守恒、ID 唯一、权重和等于总体、比例落在 [0,1]；
- 结果锚点：核心表格与冻结版本一致（快照测试）；
- 函数单元测试：自写函数的已知输入输出对。

有了测试，"大胆重构、放心重跑"才成立。我们交付给客户的分析程序全部内置测试——程序要在我们看不见的环境里由非开发者运行（"数据不出门"模式的要求），测试就是质量的远程担保。

## 第六层：文字与代码同源——报告自动化

**事故类型**：论文里的数字和代码输出对不上（手工转录错了）。

Quarto/R Markdown 把叙述与计算写在同一文档：数字、表格、图全部由代码块生成，数据更新后重新渲染，全文数字自动刷新。**报告里不存在手打的统计量**，转录错误这个物种直接灭绝。

## 采纳路线图（别贪多）

- 本周就做：第一层（脚本化）+ 第二层（Git）——成本最低、回报最大；
- 下个项目起：第三层（renv+种子）+ 第六层（Quarto）；
- 团队/长期项目：第四、五层（targets+测试）。

初期每个任务慢 20%，三个项目后由于返工锐减而净赚。可复现性不是道德负担，是**给未来的自己和同事发的一份保险**——而在我们这个行当，它已经是专业交付的入场券。
